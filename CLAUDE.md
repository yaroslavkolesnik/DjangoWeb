## Commands

All commands run from the project root (`D:\PythonWeb\pythonProject1\django\mysite`), where `manage.py` lives.

```powershell
python manage.py runserver              # Dev server (http://127.0.0.1:8000)
python manage.py makemigrations          # After model changes
python manage.py migrate                 # Apply migrations
python manage.py createsuperuser         # Admin user
python manage.py collectstatic --noinput # Build staticfiles/ for prod (WhiteNoise)
python manage.py test                    # Run all tests
python manage.py test myapp.tests.MyTest # Run a single test
python manage.py runbot                  # Start the Telegram bot (long polling, blocking)
python manage.py makemessages -l uk -l ru -l en   # Extract i18n strings
python manage.py compilemessages         # Compile .po → .mo (required after editing translations)
```

`runtime.txt` pins Python 3.12.0. Heroku deploy uses the `Procfile` (`gunicorn mysite.wsgi`).

## Environment

`.env` (gitignored) is loaded by `python-dotenv` in `mysite/settings.py`. Expected keys:
- `TELEGRAM_BOT_TOKEN` — required for `runbot`
- `SECRET_KEY`, `DEBUG`, `DATABASE_URL` — used in production (Heroku)

`requirements.txt` is **incomplete** — several runtime imports are missing (`modeltranslation`, `pyTelegramBotAPI`/`telebot`, `python-dotenv`, `django-autoslug`, `pytils`, `django-json-widget`). When adding a new install on a fresh checkout, expect to install these manually or update `requirements.txt`.

## Architecture

Django 5.2 e-commerce site with three local apps plus an embedded Telegram bot. Frontend uses Tailwind utility classes embedded in form widgets and templates.

### Apps

- **`myapp/`** — storefront. Owns `Product`, `Category`, `Brand`, `Banner`, `Order`/`OrderItem`, `Feedback`, `ContactMessage`, `ProductImage`, plus session-backed `Cart` and `Wishlist` helpers.
- **`users/`** — auth. Uses Django's built-in `LoginView`/`LogoutView`; only `register` is custom. Templates in `users/templates/users/`.
- **`bot/`** — Telegram bot. Has its own `BotMenuElem` model and a `runbot` management command (`bot/management/commands/runbot.py`) that runs `telebot.infinity_polling()`. The bot reads from `myapp.models` (Product/Category/Brand) — the storefront DB is the source of truth.

### URL routing

`mysite/urls.py` wraps `users.urls` and `myapp.urls` in `i18n_patterns`, so every user-facing URL is prefixed with the active language code (`/uk/`, `/ru/`, `/en/`). The admin is also inside `i18n_patterns`. `i18n/` (language switcher) is outside. Custom 400/404/405/500 handlers live in `myapp.views`.

Note: `myapp.views.feedback_view` hard-codes a redirect to `/ru/thanks/` — be aware this breaks for other locales.

### Cart and Wishlist (session-based)

`myapp/cart.py` and `myapp/wishlist.py` store IDs in `request.session` (no DB models). Both are exposed to all templates via `myapp.context_processors` (registered in `TEMPLATES.OPTIONS.context_processors`). The wishlist session key comes from `settings.WISHLIST_SESSION_ID`. The cart caps quantities at `Product.quantity` on add; checkout in `myapp.views.order` decrements stock and clears the cart.

### Internationalization

Three languages: `uk` (default), `ru`, `en`. Two layers:

1. **UI strings** — `gettext_lazy` / `{% trans %}`. Locale files under `myapp/locale/<lang>/LC_MESSAGES/`. Run `compilemessages` after editing `.po`.
2. **Model field translation** — `django-modeltranslation`. Translated fields are declared in `myapp/translation.py` for `Category.name`, `Brand.name`, `Product.name`, `Banner.title`/`text`. The DB has `*_uk`, `*_ru`, `*_en` columns generated by migrations (see `0020_banner_text_en_...`). After adding a field to `translation.py`, run `makemigrations` + `migrate`. Default/fallback language is `uk` (`MODELTRANSLATION_DEFAULT_LANGUAGE`).

### Product model specifics

- `slug` is auto-generated via `AutoSlugField` with `pytils.translit.slugify` (Cyrillic-aware) and `always_update=True` — slugs change when names change, which can break inbound links.
- `specifications` is a `JSONField` rendered with `JSONEditorWidget` in admin. The storefront filter UI in `myapp.views.index` reads query params prefixed `spec_<key>` and translates them to `specifications__<key>__in=[...]` lookups, so the filter works for arbitrary keys without code changes.
- `published` + `published_at` are kept in sync in `Product.save()`; `published_at` is auto-set on first publish and cleared on unpublish.
- `get_sale_percent()` and `get_average_rating()` are read in templates — preserve their signatures.

### Admin

`ProductAdmin` (`myapp/admin.py`) extends `TabbedTranslationAdmin` and adds a custom `import-csv/` URL via `get_urls()`. The CSV importer at `import_csv_view` accepts `;` or `,` delimiters, falls back from `utf-8-sig` to `cp1251` for legacy exports, and uses `update_or_create` keyed on the `id` column when present. The matching template is `myapp/templates/myapp/csv_import.html`. There is also an `export_csv` admin action.

### Templates

`myapp/templates/myapp/base.html` is the global layout used by storefront pages. Custom error pages (`400.html`, `404.html`, `405.html`, `500.html`) live next to it and are wired via `handler*` in `mysite/urls.py`. Custom template tags in `myapp/templatetags/myapp_extras.py`.

### Static and media

- `STATICFILES_DIRS = [BASE_DIR / 'static']`, `STATIC_ROOT = BASE_DIR / 'staticfiles'`.
- WhiteNoise serves static in production via `CompressedManifestStaticFilesStorage` — always run `collectstatic` before deploying.
- `MEDIA_ROOT = BASE_DIR / 'media'`. Media is only served by Django when `DEBUG=True` (see `mysite/urls.py`).

### Database

Default is SQLite (`db.sqlite3` is gitignored but currently tracked in this repo). If `DATABASE_URL` is set, `dj_database_url.parse(...)` switches to Postgres. `django_heroku.settings(locals())` runs at the bottom of `settings.py` and may override DB/static config in Heroku environments — be aware when debugging settings precedence.
